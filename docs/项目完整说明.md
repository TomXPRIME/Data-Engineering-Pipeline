## 1. 项目概述

### 1.1 项目目标

本项目构建了一个**生产级别的 SPX 500 指数数据管道**，模拟了从原始金融数据采集到最终分析仪表板的完整数据工程流程。管道采用业界标准的 **Medallion 架构**（Bronze → Silver → Gold）结合 **星型模型（Star Schema）** 设计，最终通过一个 6 标签页的 Bloomberg 风格 Streamlit 仪表板展示分析结果。

### 1.2 技术栈

| 类别 | 技术 |
|------|------|
| 数据库引擎 | DuckDB（列式 OLAP 数据库） |
| 数据处理 | Pandas |
| 文本处理 | pypdf（PDF 提取）、textblob（情感分析） |
| 文件监控 | Watchdog |
| 前端展示 | Streamlit |
| 测试框架 | pytest + 内置 unittest |
| Python 环境 | conda 环境 `qf5214_project` |
| Python 路径 | `C:/miniconda3/envs/qf5214_project/python.exe` |

### 1.3 数据规模一览

| 数据类型 | 文件数 | 数据量 | 时间跨度 | 覆盖范围 |
|----------|--------|--------|----------|----------|
| 价格数据（OHLCV） | 1 个 CSV | ~432 万行 | 2004-2024（约 20 年） | 818 只股票，5284 个交易日 |
| 财务数据（Fundamentals） | 5,726 个 CSV | 年度 + 季度报表 | 多年度 | SPX 成分股公司 |
| 财报电话会议记录（Transcripts） | 32,036 个 PDF | ~1.2 GB | 20 年 | SPX 成分股公司 |
| 股票代码参考数据 | 1 个 CSV | 947 条记录 | — | 历史上所有 SPX 成分股 |

### 2.2 架构设计理念

本项目融合了两种数据架构范式：

- **Medallion 架构**（Bronze → Silver → Gold）：负责数据质量的递进提升
  - **Bronze（青铜层）**：原始数据的原样加载，不做任何修改，保留完整审计追踪
  - **Silver（白银层）**：数据清洗、去重、质量校验、格式标准化（Parquet）
  - **Gold（黄金层）**：面向分析的数据建模，星型模型 + 预计算

- **星型模型（Star Schema）**：Gold 层内部的组织方式
  - **维度表（Dimension）**：描述"谁、何时、什么"（如股票代码、日期）
  - **事实表（Fact）**：描述"发生了多少"（如价格、财务指标、情感分数）

## 3. 原始数据详解

### 3.1 价格数据 — `data/price/spx_20yr_ohlcv_data.csv`

这是整个管道最核心的数据集，包含 SPX 500 成分股的日级别 OHLCV（开盘价、最高价、最低价、收盘价、调整收盘价、成交量）数据。

| 字段 | 类型 | 说明 |
|------|------|------|
| `ticker` | string | 股票代码（如 AAPL、MSFT） |
| `date` | string | 交易日期（YYYY-MM-DD 格式） |
| `open` | float | 开盘价 |
| `high` | float | 当日最高价 |
| `low` | float | 当日最低价 |
| `close` | float | 收盘价 |
| `adj_close` | float | 调整收盘价（考虑分红、拆股等） |
| `volume` | int | 成交量 |

**数据特点**：818 只股票 × 5284 个交易日 ≈ 432 万行。这是一个**非平衡面板数据**——不同股票上市/退市时间不同，存在缺失值。

### 3.2 财务数据 — `data/fundamental/SPX_Fundamental_History/`

包含 5,726 个 CSV 文件，每只股票有独立的文件，涵盖年度（annual）和季度（quarterly）两种频率的财务报表。

每个文件的命名格式为：`{TICKER}_{report_type}_{freq}.csv`（如 `AAPL_income_annual.csv`）。

报表类型包括：
- **income（利润表）**：revenue、net_income、eps 等
- **balance（资产负债表）**：total_assets、total_liabilities 等
- **cashflow（现金流量表）**：经营/投资/筹资活动现金流
- **profile（公司画像）**：company_name、sector、industry 等

### 3.3 PDF 财报电话会议记录 — `data/transcript/SPX_20yr_PDF_Library_10GB/`

32,036 个 PDF 文件，总大小约 1.2 GB。每份 PDF 包含一只股票在某个财报发布日的电话会议文字记录。

**文件命名格式**：`{TICKER}_Earnings_Call_{YYYY-MM-DD}.pdf`

这些 PDF 在管道中被提取文本后用于情感分析（使用 TextBlob 计算极性和主观性分数）。

### 3.4 股票代码参考数据 — `data/reference/tickers.csv`

947 条记录，记录了历史上所有曾作为 SPX 500 成分股的股票代码及其基本信息（公司名称、行业、板块等）。

## 4. 核心模块一：数据提供层（DataProvider API）

**文件位置**: `pipeline/data_provider.py`

### 4.1 设计目的

`SPXDataProvider` 类的核心作用是**模拟一个外部金融数据 API**（如 Yahoo Finance、Bloomberg）。所有下游模块都通过这个类来获取数据，而不是直接读取文件系统。这种设计实现了：

1. **解耦**：下游模块不关心数据存在哪里
2. **可测试性**：可以轻松 Mock 或替换
3. **延迟加载**：数据只在需要时读入内存
4. **时间旅行**：通过 `cutoff_date` 参数实现 Bloomberg 风格的"历史某个时间点能看到什么"的查询

### 4.2 核心类：`SPXDataProvider`

```python
class SPXDataProvider:
    """模拟金融数据 API，从本地 CSV/PDF 文件懒加载数据"""
```

#### 主要方法

| 方法 | 签名 | 返回值 | 说明 |
|------|------|--------|------|
| `get_price()` | `(ticker: str, date: str)` | `pd.DataFrame` | 获取某股票某日的 OHLCV 数据。如果日期是周末/假日或无数据，返回空 DataFrame。未知 ticker 抛出 `ValueError` |
| `get_fundamentals()` | `(ticker: str, freq="quarterly", cutoff_date=None)` | `dict` | 获取财务数据。返回字典含 `income`、`balance`、`cashflow`、`profile` 四个 DataFrame。`cutoff_date` 实现 Bloomberg 风格的"截至某个日期"过滤 |
| `get_transcript()` | `(ticker: str, date: str)` | `bytes` | 获取 PDF 原始字节。文件不存在抛 `FileNotFoundError`，读取失败抛 `DataIntegrityError` |
| `list_transcripts()` | `(ticker=None, year=None)` | `list[dict]` | 列出所有电话会议记录，可选按 ticker 或年份过滤 |
| `get_trading_dates()` | `(start_date: str, end_date: str)` | `list[str]` | 获取日期范围内的所有交易日列表 |
| `get_ticker_list()` | `()` | `list[str]` | 返回所有股票代码的排序列表 |

#### 自定义异常

```python
class DataIntegrityError(ValueError):
    """数据完整性异常——当 PDF 文件损坏或无法读取时抛出"""
```

### 4.3 数据路径常量

模块顶部定义了所有数据路径：

```python
DATA_DIR = os.path.join(ROOT_DIR, "data")
PRICE_CSV = os.path.join(DATA_DIR, "price", "spx_20yr_ohlcv_data.csv")
FUNDAMENTAL_DIR = os.path.join(DATA_DIR, "fundamental", "SPX_Fundamental_History")
TRANSCRIPT_DIR = os.path.join(DATA_DIR, "transcript", "SPX_20yr_PDF_Library_10GB")
TICKERS_CSV = os.path.join(DATA_DIR, "reference", "tickers.csv")
```

### 4.4 `cutoff_date` 参数详解

这是 Phase 1 的核心功能。当调用 `get_fundamentals(ticker, freq, cutoff_date="2020-06-30")` 时：

- 只返回在 `2020-06-30` 这个日期**之前已经发布**的财务数据
- 模拟了真实的金融场景：你在 2020 年 6 月是看不到 2020 年 Q3 的财报的
- 这对于回测策略至关重要，避免了**前视偏差（look-ahead bias）**

## 5. 核心模块二：虚拟时钟模拟器（ComprehensiveSimulator）

**文件位置**: `pipeline/simulators/comprehensive_simulator.py`

### 5.1 设计目的

在真实生产环境中，数据是**随着时间不断到达**的。模拟器通过一个"虚拟时钟"模拟这个过程——时钟从一个交易日前进到下一个交易日，每次调用 DataProvider 获取当天数据，然后写入 `output/landing_zone/` 目录。

### 5.2 核心类：`ComprehensiveSimulator`

```python
class ComprehensiveSimulator:
    """虚拟时钟模拟器——按交易日推进，发射数据到 landing_zone"""
```

#### 初始化时的关键操作

1. **构建电话会议索引**（`_transcript_index`）：按日期组织所有 PDF 文件路径，方便快速查找
2. **构建财务数据索引**（`_fundamental_index`）：按财务日期组织所有 CSV 文件路径
3. **种子化所有财务文件**（`_seed_all_fundamentals()`）：在回退模式开始时，一次性将所有财务文件复制到 `landing_zone/fundamentals/{ticker}/` 目录
4. **读取水位线**（watermark）：支持断点续传，水位线存储在 `output/.watermark`

#### 主要方法

| 方法 | 签名 | 说明 |
|------|------|------|
| `run_backfill()` | `(start_date, end_date, delay=0)` | **回退模式**——遍历指定日期范围内的所有交易日，批量加载历史数据 |
| `run_realtime()` | `(start_date, delay=1.0)` | **实时模式**——从指定日期开始持续运行（带时间延迟，模拟真实数据流） |
| `_emit_price()` | `(date)` | 将某日所有股票的价格数据写入 `landing_zone/prices/price_YYYY-MM-DD.csv` |
| `_emit_fundamentals()` | `(date)` | 将某日到期的财务文件复制到 `landing_zone/fundamentals/{ticker}/` |
| `_emit_transcripts()` | `(date)` | 将某日的 PDF 复制到 `landing_zone/transcripts/` |
| `_seed_all_fundamentals()` | `()` | 一次性将所有财务文件按 ticker 分区复制到 landing_zone |
| `_enqueue_messages()` | `(date, data_type, ticker)` | 将事件写入 `queue_messages` 表，供事件驱动的下游消费 |

#### 运行模式对比

| 模式 | 用途 | 日期推进 | 延迟 |
|------|------|----------|------|
| `backfill` | 批量加载历史数据 | 遍历 start→end 所有交易日 | 可设为 0 |
| `realtime` | 模拟实时数据流 | 从 start 持续运行 | 默认 1 秒/日 |

### 5.3 输出格式

模拟器产生的文件遵循严格的命名规范：

```
output/landing_zone/
├── prices/
│   └── price_2024-01-02.csv        # 某日所有股票的价格
├── fundamentals/
│   └── AAPL/                       # 按 ticker 分区
│       └── AAPL_income_annual.csv  # 具体财务文件
└── transcripts/
    └── AAPL_Earnings_Call_2024-01-15.pdf
```

## 6. 核心模块三：青铜层摄入引擎（IngestionEngine）

**文件位置**: `pipeline/ingestion_engine.py`  
**数据库**: `duckdb/spx_analytics.duckdb`

### 6.1 设计目的

IngestionEngine 负责将 `landing_zone/` 中的原始文件**加载到 DuckDB 数据库**中，形成 Bronze 层。关键设计原则是**原样加载（load as-is）**——不做任何数据清洗或转换，保留完整的审计追踪。

### 6.2 三种运行模式

| 模式 | 命令 | 说明 |
|------|------|------|
| `scan` | `python -m pipeline.ingestion_engine --mode scan` | **扫描模式**——一次性扫描 landing_zone 中的所有文件，适合回退 |
| `watch` | `python -m pipeline.ingestion_engine --mode watch` | **监控模式**——使用 Watchdog 持续监控目录变化，新文件到达时自动摄入 |
| `queue` | `python -m pipeline.ingestion_engine --mode queue` | **队列模式**——轮询 `queue_messages` 表中的 PENDING 消息，按事件驱动处理 |

### 6.3 Bronze 层表结构

Bronze 层包含 5 张表，定义在 `duckdb/create_bronze_tables.sql` 中：

#### 6.3.1 `raw_price_stream` — 价格数据流

| 列名 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 自增主键 |
| `ticker` | VARCHAR | 股票代码 |
| `date` | DATE | 交易日期 |
| `open`, `high`, `low`, `close`, `adj_close` | DOUBLE | 价格字段 |
| `volume` | BIGINT | 成交量 |
| `received_at` | TIMESTAMP | 数据到达时间 |

#### 6.3.2 `raw_fundamental_index` — 财务数据索引

| 列名 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 自增主键 |
| `ticker` | VARCHAR | 股票代码 |
| `report_type` | VARCHAR | 报表类型（income/balance/cashflow） |
| `freq` | VARCHAR | 频率（annual/quarterly） |
| `fiscal_date` | DATE | 财务日期 |
| `file_path` | VARCHAR | 文件路径 |
| `received_at` | TIMESTAMP | 数据到达时间 |

#### 6.3.3 `raw_transcript_index` — 电话会议索引

| 列名 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 自增主键 |
| `ticker` | VARCHAR | 股票代码 |
| `event_date` | DATE | 电话会议日期 |
| `pdf_path` | VARCHAR | PDF 文件路径 |
| `text_hash` | VARCHAR | 文本哈希（用于去重） |
| `received_at` | TIMESTAMP | 数据到达时间 |

#### 6.3.4 `ingestion_audit` — 审计追踪

记录每次摄入操作的结果（成功/失败）、文件哈希、错误信息等。

#### 6.3.5 `queue_messages` — 消息队列

用于事件驱动的摄入模式。消息状态包括：`PENDING`（待处理）、`DONE`（已完成）、`FAILED`（失败）。

### 6.4 摄入方法

| 方法 | 说明 |
|------|------|
| `ingest_price_file(filepath)` | 读取价格 CSV，批量插入到 `raw_price_stream`。使用事务保证原子性 |
| `ingest_fundamental_file(filepath, ticker)` | 从文件名提取元数据（报表类型、频率、财务日期），写入 `raw_fundamental_index` |
| `ingest_transcript_file(filepath)` | 从文件名提取 ticker 和日期，写入 `raw_transcript_index` |
| `scan_and_ingest()` | 扫描三个 landing_zone 子目录并摄入所有文件 |
| `poll_queue(batch_size)` | 处理最多 `batch_size` 条 PENDING 消息 |

### 6.5 辅助函数

- `_safe_float(val)`：将 NaN/None 安全转换为 SQL NULL 的浮点数
- `_safe_int(val)`：将 NaN/None 安全转换为 SQL NULL 的整数
- `LandingZoneHandler`：Watchdog 的文件系统事件处理器，响应新文件的创建和修改

## 7. 核心模块四：ELT 管道（Bronze → Silver）

**文件位置**: `pipeline/elt_pipeline.py`

### 7.1 设计目的

ELT（Extract-Load-Transform）管道将 Bronze 层的原始数据**转换为干净的、标准化的 Silver 层 Parquet 文件**。这是数据质量提升的关键环节。

### 7.2 运行方式

通过 `--resource` 参数指定要转换的资源类型：

```bash
python -m pipeline.elt_pipeline --resource all          # 转换所有资源
python -m pipeline.elt_pipeline --resource price        # 仅转换价格数据
python -m pipeline.elt_pipeline --resource fundamentals # 仅转换财务数据
python -m pipeline.elt_pipeline --resource transcripts  # 仅转换电话会议
python -m pipeline.elt_pipeline --resource sentiment    # 仅做情感分析
```

### 7.3 四类转换详解

#### 7.3.1 价格数据转换 — `transform_price()`

**输入**: Bronze 表 `raw_price_stream`  
**输出**: `output/silver/price/date=YYYY-MM-DD/*.parquet`（Hive 分区格式）

**处理步骤**：
1. **去重**：对每个 `(ticker, date)` 组合，保留 `received_at` 最新的一条记录（模拟真实场景中数据可能被多次修正）
2. **质量校验**：执行 11 条质量规则（见下表），违规行**不被删除**，而是记录到 `silver_quality_issues` 表中
3. **导出**：按日期分区写入 Parquet 文件

**11 条质量规则**：

| 规则名 | 检查条件 | 业务含义 |
|--------|----------|----------|
| `price_positive` | open/high/low/close/adj_close > 0 | 价格必须为正数 |
| `volume_non_negative` | volume >= 0 | 成交量不能为负 |
| `high_gte_low` | high >= low | 最高价不能低于最低价 |
| `high_gte_open` | high >= open | 最高价不能低于开盘价 |
| `high_gte_close` | high >= close | 最高价不能低于收盘价 |
| `low_lte_open` | low <= open | 最低价不能高于开盘价 |
| `low_lte_close` | low <= close | 最低价不能高于收盘价 |

#### 7.3.2 财务数据转换 — `transform_fundamentals()`

**输入**: Bronze 表 `raw_fundamental_index` + 原始 CSV 文件  
**输出**: `output/silver/fundamentals/ticker=XXX/data.parquet`

**处理步骤**：
1. **去重**：从 `raw_fundamental_index` 中选取最新的索引记录
2. **读取 CSV**：通过 DataProvider 读取原始财务数据
3. **EAV 转换**：将宽表（每列一个财务指标）转换为**长格式（Entity-Attribute-Value）**，即 `(ticker, fiscal_date, metric_name, metric_value, freq)` 格式
4. **频率传播**：确保 `freq` 字段（annual/quarterly）被正确保留
5. **导出**：按 ticker 分区写入 Parquet

#### 7.3.3 电话会议文本提取 — `transform_transcripts()`

**输入**: Bronze 表 `raw_transcript_index` + PDF 文件  
**输出**: `output/silver/transcript_text/ticker=XXX/date=YYYY-MM-DD/content.txt`

**处理步骤**：
1. **去重**：选取最新的索引记录
2. **PDF 文本提取**：使用 `pypdf` 读取 PDF 并提取文本内容
3. **导出**：按 ticker 和日期分区写入纯文本文件

#### 7.3.4 情感分析 — `transform_sentiment()`

**输入**: `output/silver/transcript_text/` 下的 `.txt` 文件  
**输出**: `output/silver/transcript_sentiment/ticker=XXX/event_date=YYYY-MM-DD/*.parquet`

**处理步骤**：
1. 读取每个 `content.txt` 文件
2. 使用 **TextBlob** 计算两个情感指标：
   - **polarity（极性）**：-1.0（极度负面）到 +1.0（极度正面），0 为中性
   - **subjectivity（主观性）**：0.0（客观事实）到 1.0（主观观点）
3. 将情感分数与原始文本一起写入 Parquet 文件

## 8. 核心模块五：Gold 星型模型层

### 8.1 整体架构

Gold 层是管道的**最终分析层**，采用星型模型组织数据。构建过程由 `gold/build_gold_layer.py` 驱动，依次执行三个 SQL 文件：

```
1. gold/sql/create_star_schema.sql   → 5 张核心表（2 维度 + 3 事实）
2. gold/sql/create_materialized.sql  → 3 张物化分析表
3. gold/sql/create_olap_views.sql    → 7 个 OLAP 视图
```

所有数据存储在 `duckdb/spx_analytics.duckdb` 中。

### 8.2 维度表生成器

#### 8.2.1 `dim_date` — 日期维度表

**生成器**: `gold/dim_date_generator.py`  
**数据范围**: 2000-01-01 至 2100-12-31

**Schema**:

| 列名 | 类型 | 说明 |
|------|------|------|
| `date` | DATE | 日期 |
| `year` | INT | 年份 |
| `quarter` | INT | 季度（1-4） |
| `month` | INT | 月份（1-12） |
| `day` | INT | 日（1-31） |
| `day_of_week` | INT | 星期几（0=周一） |
| `is_trading_day` | BOOLEAN | 是否为交易日 |
| `is_holiday` | BOOLEAN | 是否为假日 |
| `holiday_name` | VARCHAR | 假日名称 |
| `trading_day_offset` | INT | 距上一个交易日的偏移 |

**美国股市假日**：新年、马丁·路德·金日、总统日、阵亡将士纪念日、独立日、劳动节、感恩节、圣诞节（共 8 个）。

#### 8.2.2 `dim_ticker` — 股票代码维度表（SCD Type 2）

**生成器**: `gold/dim_ticker_generator.py`

**SCD Type 2（缓慢变化维度类型 2）** 的含义：当一只股票的行业/板块分类发生变化时，不是更新原有记录，而是**关闭旧记录**（设置 `valid_to`）并**创建新记录**（设置新的 `valid_from`）。这样可以追溯历史上任意时间点一只股票属于什么板块。

**Schema**:

| 列名 | 类型 | 说明 |
|------|------|------|
| `ticker` | VARCHAR | 股票代码 |
| `company_name` | VARCHAR | 公司名称 |
| `sector` | VARCHAR | 板块分类 |
| `industry` | VARCHAR | 行业分类 |
| `valid_from` | DATE | 该记录有效期的起始日期 |
| `valid_to` | DATE | 该记录有效期的结束日期 |
| `is_current` | BOOLEAN | 是否为当前有效记录 |

### 8.3 星型模型核心表（5 张）

#### 8.3.1 `dim_date` 和 `dim_ticker`

见上方维度表生成器部分。

#### 8.3.2 `fact_daily_price` — 日价格事实表

**来源**: Silver 价格 Parquet + 窗口函数  
**数据量**: ~205,318 行

| 列名 | 说明 |
|------|------|
| `ticker`, `date` | 外键关联维度表 |
| `open`, `high`, `low`, `close`, `adj_close`, `volume` | OHLCV 数据 |
| `daily_return` | 日收益率 = (close - prev_close) / prev_close |
| `next_1d_return` | 次日收益率 |
| `next_5d_return` | 后 5 日收益率 |

#### 8.3.3 `fact_quarterly_fundamentals` — 季度财务事实表

**来源**: Silver 财务 Parquet（经过行转列/透视处理）  
**数据量**: ~4,504 行

| 列名 | 说明 |
|------|------|
| `ticker`, `fiscal_date` | 股票代码和财务日期 |
| `report_type` | 报表类型 |
| `freq` | annual / quarterly |
| `revenue` | 营业收入 |
| `net_income` | 净利润 |
| `total_assets` | 总资产 |
| `total_liabilities` | 总负债 |
| `eps` | 每股收益 |
| `book_value_per_share` | 每股净资产 |

#### 8.3.4 `fact_earnings_transcript` — 财报电话会议情感事实表

**来源**: Silver 情感分析 Parquet + 价格事实表 JOIN  
**数据量**: ~1,954 行

| 列名 | 说明 |
|------|------|
| `ticker`, `event_date` | 股票代码和电话会议日期 |
| `sentiment_polarity` | 情感极性分数（-1 到 +1） |
| `sentiment_subjectivity` | 主观性分数（0 到 1） |
| `close_on_event` | 电话会议当日收盘价 |
| `next_1d_return` | 会后 1 日收益率 |
| `next_5d_return` | 会后 5 日收益率 |

### 8.4 物化事实表（3 张）

这些表通过**预计算复杂的窗口函数**来加速查询，避免每次查询都重新计算。

#### 8.4.1 `fact_rolling_volatility` — 滚动波动率表

**计算方式**: 20 日/60 日滚动窗口计算年化波动率  
**数据量**: ~147,003 行

| 列名 | 说明 |
|------|------|
| `ticker`, `date`, `close` | 基础字段 |
| `annualized_vol_20d` | 20 日年化波动率 |
| `annualized_vol_60d` | 60 日年化波动率 |
| `annualized_return_20d` | 20 日年化收益率 |

#### 8.4.2 `fact_momentum_signals` — 动量信号表

**计算方式**: 多周期动量 + 均线偏离度  
**数据量**: ~112,705 行

| 列名 | 说明 |
|------|------|
| `ticker`, `date`, `close` | 基础字段 |
| `momentum_5d`, `momentum_20d`, `momentum_60d` | 5/20/60 日动量 |
| `dist_pct_from_ma20` | 距 20 日均线的偏离百分比 |
| `dist_pct_from_ma60` | 距 60 日均线的偏离百分比 |
| `trend_signal` | 趋势信号值（见下表） |

**趋势信号值**：

| 值 | 含义 |
|----|------|
| `STRONG_UPTREND` | 强上升趋势 |
| `WEAK_UPTREND` | 弱上升趋势 |
| `NEUTRAL` | 中性 |
| `WEAK_DOWNTREND` | 弱下降趋势 |
| `STRONG_DOWNTREND` | 强下降趋势 |

#### 8.4.3 `fact_ar1_results` — AR(1) 回归结果表

**计算方式**: AR(1) 自回归模型 OLS 估计（r_t = α + β × r_{t-1}），60 行滚动窗口  
**数据量**: ~135,159 行

| 列名 | 说明 |
|------|------|
| `ticker`, `date`, `close` | 基础字段 |
| `daily_return` | 日收益率 |
| `alpha_ar1` | AR(1) 截距项（α） |
| `beta_ar1` | AR(1) 自回归系数（β） |
| `r_squared_ar1` | R² 拟合优度 |
| `n_obs` | 窗口内观测值数量 |

### 8.5 OLAP 视图（7 个）

视图是**轻量级的虚拟表**，不存储数据，仅在查询时动态计算。

| 视图名 | 依赖表 | 行数 | 用途 |
|--------|--------|------|------|
| `v_market_daily_summary` | fact_daily_price | ~251 | 每日市场汇总：平均收盘价、平均收益率、总成交量 |
| `v_ticker_profile` | fact_daily_price + dim_ticker | ~818 | 每只股票最新快照：公司名称、板块、最新收盘价 |
| `v_fundamental_snapshot` | fact_quarterly_fundamentals | ~595 | 每只股票最新财务快照 |
| `v_fundamental_history` | fact_quarterly_fundamentals + fact_daily_price | ~205,318 | **Bloomberg AS OF 风格**：在每个价格日期，显示当时能看到的最新财务数据 |
| `v_sentiment_price_view` | fact_earnings_transcript | ~1,954 | 情感分数 + 前后收益率 |
| `v_sentiment_binned_returns` | fact_earnings_transcript | ~2 | 情感分桶（正面/中性/负面，阈值 ±0.2）与远期收益率的关系 |
| `v_sector_rotation` | fact_daily_price + dim_ticker | ~52 | 季度板块轮动排名 |

#### `v_fundamental_history` 详解——Bloomberg AS OF 风格

这是最复杂的视图之一。它模拟了这样一个场景：**"在 2024 年 3 月 15 日这天，一个分析师能看到哪些已经发布的财务数据？"**

实现方式：将价格事实表与财务事实表进行 AS OF JOIN——对于每个价格日期，找到该日期之前最新发布的财务数据。这避免了回测中的前视偏差。

### 8.6 Gold 层构建流程

`gold/build_gold_layer.py` 的 `GoldLayerBuilder` 类执行以下步骤：

1. **检查 Silver Parquet 文件是否存在**——缺少任何一个关键文件都会报错退出
2. **验证 SQL 文件存在**——确保三个 SQL 文件都存在
3. **切换工作目录到项目根目录**——因为 SQL 中的 Parquet 读取使用相对路径
4. **连接 DuckDB**——打开 `duckdb/spx_analytics.duckdb`
5. **依次执行 SQL 文件**——按 star_schema → materialized → olap_views 顺序
6. **验证所有表有数据**——检查每张表的行数
7. **打印样本数据和汇总表**——在控制台输出验证结果

### 9.1 设计目的

查询层是 Gold 层和 Streamlit 仪表板之间的**中间件**。它封装了所有 SQL 查询逻辑，提供了：

### 9.2 GoldDataProvider

```python
class GoldDataProvider:
    """DuckDB 连接的上下文管理器"""
    
    def execute(self, query: str, params: tuple = None) -> pd.DataFrame:
        """执行参数化 SQL 查询，返回 DataFrame"""
```

使用方式：
```python
with GoldDataProvider() as db:
    result = db.execute("SELECT * FROM fact_daily_price WHERE ticker = ?", (ticker,))
```

| `gold/tests/test_dim_ticker_generator.py` | pytest 单元测试 | 6 项 | SCD Type 2 结构、板块变化、行业变化、无变化单行、valid_to 逻辑、is_current 标记 |

### 12.2 端到端测试详解（`test_pipeline.py`）

`test_pipeline.py` 是整个管道最重要的集成测试。它依次执行 6 个子进程调用：

1. **清理**：删除旧的 `output/` 和 `duckdb/spx_analytics.duckdb`
2. **初始化 Bronze**：创建 DuckDB 表结构
3. **模拟器**：生成 2024 年 1 月的 landing_zone 文件
4. **摄入**：将 landing_zone 文件加载到 Bronze
5. **ELT**：转换所有四种资源（价格、财务、电话会议、情感）
6. **Gold 构建 + 验证**：构建星型模型并运行 45 项检查

### 12.3 Gold 视图验证（`gold/tests/test_gold_views.py`）

这是最全面的验证脚本，包含 45 项检查：

- **维度表验证**：`dim_date` 和 `dim_ticker` 的存在性、行数、列名、数据类型
- **事实表验证**：`fact_daily_price`、`fact_quarterly_fundamentals`、`fact_earnings_transcript` 的存在性、行数、列名、外键关系
- **物化表验证**：`fact_rolling_volatility`、`fact_momentum_signals`、`fact_ar1_results` 的存在性、行数、列名
- **OLAP 视图验证**：7 个视图的存在性、行数、列名
- **数据质量验证**：波动率范围、收益率范围、情感分数范围等

## 13. 开发规范

详见 `STANDARDS.md` 文件。以下是核心要点：

### 13.1 代码风格

- 遵循 PEP 8 规范
- 使用 `black` 格式化（行宽 120）
- 使用 `ruff` 进行 linting

### 13.2 命名约定

| 类型 | 规范 | 示例 |
|------|------|------|
| 变量/函数 | snake_case | `get_price()`, `ticker_list` |
| 类名 | PascalCase | `SPXDataProvider`, `GoldLayerBuilder` |
| 常量 | UPPER_SNAKE_CASE | `DATA_DIR`, `PRICE_CSV` |
| 文件名 | snake_case | `data_provider.py`, `build_gold_layer.py` |
| DuckDB 表名 | snake_case | `raw_price_stream`, `fact_daily_price` |
| landing_zone 文件 | 严格格式 | `price_YYYY-MM-DD.csv` |

### 13.3 测试规范

- 单元测试使用 pytest
- 测试文件命名：`test_<模块名>.py`
- 测试函数命名：`test_<被测行为>_<预期结果>`

### 13.4 日志规范

- 使用 Python 标准 `logging` 模块
- 格式：`%(asctime)s [%(levelname)s] %(name)s: %(message)s`
- 日志级别：DEBUG（调试）、INFO（正常流程）、WARNING（可恢复问题）、ERROR（需要人工介入）

### 13.5 Git 提交规范

遵循 Conventional Commits 格式：

```
<type>(<scope>): <description>

[optional body]
```

| type | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `test` | 测试相关 |
| `refactor` | 重构（不改变功能） |

## 附录：核心概念速查表

### Medallion 架构三层对比

| 特性 | Bronze（青铜） | Silver（白银） | Gold（黄金） |
|------|---------------|---------------|-------------|
| 数据质量 | 原样加载 | 清洗去重 | 分析就绪 |
| 存储格式 | DuckDB 表 | Parquet 文件 | DuckDB 表/视图 |
| 转换程度 | 无 | 去重 + 质检 + EAV | 星型模型 + 窗口函数 |
| 用途 | 原始数据备份 | 数据清洗中间层 | 分析和查询 |

### 关键数据流

```
CSV/PDF → DataProvider → Simulator → Landing Zone → Bronze(DuckDB) → Silver(Parquet) → Gold(Star Schema) → Query Layer → Dashboard
```

所有 6 个阶段均已通过测试验证，仪表板可以正常启动并展示数据。
