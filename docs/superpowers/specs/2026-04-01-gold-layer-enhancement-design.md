## 2. Critical Bug Fixes

### 2.1 ingestion_engine.py — Batch INSERT

**Problem:** Row-by-row `INSERT` in a loop causes O(n) DB calls. With 818 tickers × 5284 days = 4.3M rows, ingestion takes hours.

**Fix:** Replace loop with `con.executemany()` for batch insertion.

```python
# BEFORE (lines 107-140)
for _, row in df.iterrows():
    con.execute("INSERT INTO raw_price_stream ...", [...])

# AFTER
con.begin()
try:
    con.executemany(
        "INSERT INTO raw_price_stream (ticker, date, open, high, low, close, adj_close, volume) VALUES (?, ?, ?, ?, ?, ?, ?, ?)",
        rows
    )
    con.commit()
except Exception as e:
    con.rollback()
    raise
```

**Files:** `pipeline/ingestion_engine.py`

### 2.2 ingestion_engine.py — Audit Log Truthfulness

**Problem:** Even if individual rows fail inside the loop, `ingest_price_file` always logs `SUCCESS`.

**Fix:** Track row-level failures, update status to `PARTIAL` or `FAILED`.

```python
failed_rows = 0
for _, row in df.iterrows():
    try:
        con.execute("INSERT ...", [...])
        rows_inserted += 1
    except Exception as e:
        failed_rows += 1
        logger.warning(f"Failed to insert row for {row.get('Ticker')}: {e}")

if failed_rows > 0 and rows_inserted == 0:
    status = "FAILED"
elif failed_rows > 0:
    status = "PARTIAL"
else:
    status = "SUCCESS"

self._log_audit("price", ticker, market_date, file_hash, status)
```

**Files:** `pipeline/ingestion_engine.py`

### 2.3 ingestion_engine.py — Module-Level Helpers

**Problem:** `safe_float()` and `safe_int()` are redefined inside the loop on every iteration, with `import math` repeated per call.

**Fix:** Move to module-level functions.

```python
# Module level (top of file)
import math

def _safe_float(val):
    """Convert NaN/None to SQL NULL."""
    if val is None or (isinstance(val, float) and math.isnan(val)):
        return None
    return val

def _safe_int(val):
    """Convert NaN/None to SQL NULL, coerce to int."""
    if val is None or (isinstance(val, float) and math.isnan(val)):
        return None
    return int(val) if val is not None else None
```

## 3. Message Broker (DuckDB-Based Queue)

### 3.1 Motivation

The current watchdog-based ingestion uses filesystem polling, which:
- Consumes CPU with repeated `stat()` calls
- Has poor scalability (can't distribute across machines)
- Introduces coupling between file-watcher and processing logic

Replacing watchdog with a message queue provides:
- **Decoupling:** Producer (Simulator) and consumer (Ingestion) communicate via queue
- **Durability:** Messages persist in DuckDB; crash during processing does not lose data
- **Testability:** Easy to inject test messages without creating files
- **Presentation value:** Demonstrates event-driven architecture for course project

### 3.2 Architecture

```
┌─────────────────────────────────────────────────────────┐
│  BEFORE (watchdog)                                      │
│                                                         │
│  Landing Zone (file) → watchdog → Ingestion Engine       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  AFTER (message queue)                                  │
│                                                         │
│  Simulator → queue_messages table → Ingestion Engine    │
│                (DuckDB)              (consumer)         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 Queue Table Schema

```sql
CREATE TABLE IF NOT EXISTS queue_messages (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    source      VARCHAR(50),          -- 'simulator', 'elt', etc.
    msg_type    VARCHAR(100),         -- 'price_file', 'fundamental_file', 'transcript_file'
    payload     VARCHAR(1000),        -- JSON: {"filepath": "...", "ticker": "AAPL", "date": "2024-01-15"}
    status      VARCHAR(20) DEFAULT 'PENDING',  -- PENDING, PROCESSING, DONE, FAILED
    created_at  TIMESTAMP DEFAULT NOW(),
    consumed_at TIMESTAMP,
    error_message VARCHAR(500)
);
```

### 3.4 Producer (Simulator Side)

After Simulator writes a file to landing_zone, it also inserts a message. Three message types:

```python
def _enqueue_message(msg_type: str, payload: dict):
    """Insert a message into the queue for downstream consumers."""
    import json
    con = get_queue_connection()
    con.execute(
        "INSERT INTO queue_messages (source, msg_type, payload, status) VALUES (?, ?, ?, 'PENDING')",
        ["simulator", msg_type, json.dumps(payload)]
    )

# Price file written:
_enqueue_message("price_file", {"filepath": str(csv_path), "ticker": ticker, "date": market_date})

# Fundamental file written:
_enqueue_message("fundamental_file", {"filepath": str(fund_csv_path), "ticker": ticker, "report_type": report_type})

# Transcript file written:
_enqueue_message("transcript_file", {"filepath": str(pdf_path), "ticker": ticker, "event_date": event_date})
```

### 3.5 Consumer (Ingestion Side)

Instead of watchdog polling, the ingestion engine polls the queue table:

```python
def poll_queue(self, batch_size: int = 100):
    """Fetch pending messages, process them, mark as done or failed."""
    con = self._get_connection()

    rows = con.execute("""
        SELECT id, msg_type, payload
        FROM queue_messages
        WHERE status = 'PENDING'
        ORDER BY created_at
        LIMIT ?
    """, [batch_size]).fetchall()

    for row in rows:
        msg_id, msg_type, payload = row
        try:
            payload_dict = json.loads(payload)
            if msg_type == "price_file":
                self.ingest_price_file(Path(payload_dict["filepath"]))
            elif msg_type == "fundamental_file":
                self.ingest_fundamental_file(Path(payload_dict["filepath"]), payload_dict["date"])
            elif msg_type == "transcript_file":
                self.ingest_transcript_file(Path(payload_dict["filepath"]))
            else:
                logger.warning(f"Unknown msg_type: {msg_type}")

            con.execute(
                "UPDATE queue_messages SET status = 'DONE', consumed_at = NOW() WHERE id = ?",
                [msg_id]
            )
        except Exception as e:
            con.execute(
                "UPDATE queue_messages SET status = 'FAILED', error_message = ? WHERE id = ?",
                [str(e), msg_id]
            )
```

### 3.6 Implementation Notes

- **No new dependencies:** Uses existing DuckDB connection, zero external packages
- **Transactional safety:** Message marked `PROCESSING` before work, `DONE` after success, `FAILED` on error
- **Idempotency:** Ingestion engine already handles duplicate file paths (UNIQUE constraint); re-processing a message is safe
- **Simulator change:** Only `comprehensive_simulator.py` needs modification (add `_enqueue_message` call)
- **Ingestion engine change:** Replace `LandingZoneHandler` watchdog with `poll_queue()` loop
- **Cleanup:** A background thread can periodically `DELETE FROM queue_messages WHERE status = 'DONE' AND consumed_at < NOW() - INTERVAL '7 days'`

### 4.2 New View: v_rolling_volatility

**Purpose:** 20-day and 60-day rolling volatility (annualized), plus annualized return.

**SQL:**
```sql
CREATE OR REPLACE VIEW v_rolling_volatility AS
WITH daily_returns AS (
    SELECT
        ticker, date, close,
        (close - LAG(close, 1) OVER (PARTITION BY ticker ORDER BY date))
            / NULLIF(LAG(close, 1) OVER (PARTITION BY ticker ORDER BY date), 0) AS daily_return
    FROM silver_price
),
vol AS (
    SELECT
        ticker, date, close, daily_return,
        STDDEV(daily_return) OVER w20 * SQRT(252) AS annualized_vol_20d,
        STDDEV(daily_return) OVER w60 * SQRT(252) AS annualized_vol_60d,
        AVG(daily_return) OVER w20 * 252 AS annualized_return_20d
    FROM daily_returns
    WINDOW w20 AS (PARTITION BY ticker ORDER BY date ROWS BETWEEN 19 PRECEDING AND CURRENT ROW),
           w60 AS (PARTITION BY ticker ORDER BY date ROWS BETWEEN 59 PRECEDING AND CURRENT ROW)
)
SELECT ticker, date, close,
    ROUND(annualized_vol_20d, 6) AS annualized_vol_20d,
    ROUND(annualized_vol_60d, 6) AS annualized_vol_60d,
    ROUND(annualized_return_20d, 6) AS annualized_return_20d
FROM vol
WHERE annualized_vol_20d IS NOT NULL
ORDER BY ticker, date;
```

**Columns:** `ticker, date, close, annualized_vol_20d, annualized_vol_60d, annualized_return_20d`

### 4.3 New View: v_momentum_signals

**Purpose:** Multi-period momentum (5d, 20d, 60d) + moving average deviation + trend signal classification.

**SQL:**
```sql
CREATE OR REPLACE VIEW v_momentum_signals AS
WITH price_analytics AS (
    SELECT
        ticker, date, close,
        LAG(close, 5) OVER w AS lag5,
        LAG(close, 20) OVER w AS lag20,
        LAG(close, 60) OVER w AS lag60,
        AVG(close) OVER w20 AS ma20,
        AVG(close) OVER w60 AS ma60
    FROM silver_price
    WINDOW w AS (PARTITION BY ticker ORDER BY date),
           w20 AS (PARTITION BY ticker ORDER BY date ROWS BETWEEN 19 PRECEDING AND CURRENT ROW),
           w60 AS (PARTITION BY ticker ORDER BY date ROWS BETWEEN 59 PRECEDING AND CURRENT ROW)
)
SELECT
    ticker, date, close,
    ROUND((close - lag5) / NULLIF(lag5, 0), 6) AS momentum_5d,
    ROUND((close - lag20) / NULLIF(lag20, 0), 6) AS momentum_20d,
    ROUND((close - lag60) / NULLIF(lag60, 0), 6) AS momentum_60d,
    ROUND((close - ma20) / NULLIF(ma20, 0), 6) AS dist_pct_from_ma20,
    ROUND((close - ma60) / NULLIF(ma60, 0), 6) AS dist_pct_from_ma60,
    CASE
        WHEN close > ma20 AND close > ma60 THEN 'STRONG_UPTREND'
        WHEN close < ma20 AND close < ma60 THEN 'STRONG_DOWNTREND'
        WHEN close > ma20 THEN 'WEAK_UPTREND'
        WHEN close < ma20 THEN 'WEAK_DOWNTREND'
        ELSE 'NEUTRAL'
    END AS trend_signal
FROM price_analytics
WHERE lag5 IS NOT NULL AND lag20 IS NOT NULL AND lag60 IS NOT NULL
ORDER BY ticker, date;
```

**Columns:** `ticker, date, close, momentum_5d, momentum_20d, momentum_60d, dist_pct_from_ma20, dist_pct_from_ma60, trend_signal`

### 4.4 New View: v_sector_rotation

**Purpose:** Quarterly sector-level performance ranking to identify sector rotation patterns.

> **Note:** `silver_fundamentals` stores sector as key-value pairs (metric='sector', value='Information Technology'), NOT as a `gics_sector` column. The pivot pattern is required.

**SQL:**
```sql
CREATE OR REPLACE VIEW v_sector_rotation AS
WITH sector_daily AS (
    SELECT
        MAX(CASE WHEN f.metric = 'sector' THEN f.value END) AS sector,
        p.date,
        EXTRACT(YEAR FROM p.date) AS year,
        EXTRACT(QUARTER FROM p.date) AS quarter,
        AVG(p.close) AS avg_close,
        SUM(p.volume) AS total_volume,
        STDDEV(p.close) AS price_std,
        COUNT(DISTINCT p.ticker) AS ticker_count
    FROM silver_price p
    LEFT JOIN silver_fundamentals f ON p.ticker = f.ticker
    WHERE f.metric = 'sector' OR f.metric IS NULL
    GROUP BY sector, p.date, year, quarter
),
sector_quarterly AS (
    SELECT
        sector,
        year,
        quarter,
        AVG(avg_close) AS avg_close,
        SUM(total_volume) AS total_volume,
        AVG(price_std) AS avg_volatility,
        AVG(ticker_count) AS avg_ticker_count,
        RANK() OVER (
            PARTITION BY year, quarter
            ORDER BY (AVG(avg_close) - LAG(AVG(avg_close)) OVER (PARTITION BY sector ORDER BY year, quarter))
                     / NULLIF(LAG(AVG(avg_close)) OVER (PARTITION BY sector ORDER BY year, quarter), 0)
                     DESC
        ) AS momentum_rank
    FROM sector_daily
    GROUP BY sector, year, quarter
)
SELECT * FROM sector_quarterly
ORDER BY year, quarter, momentum_rank;
```

**Columns:** `sector, year, quarter, avg_close, total_volume, avg_volatility, avg_ticker_count, momentum_rank`

### 4.5 New View: v_sentiment_binned_returns

**Purpose:** Aggregate returns grouped by sentiment bucket (POSITIVE / NEUTRAL / NEGATIVE) to analyze whether earnings call sentiment predicts short-term price movement.

**SQL:**
```sql
CREATE OR REPLACE VIEW v_sentiment_binned_returns AS
WITH sentiment_returns AS (
    SELECT
        s.ticker,
        s.event_date,
        s.sentiment_polarity,
        s.sentiment_subjectivity,
        p.next_1d_return,
        p.next_5d_return
    FROM silver_sentiment s
    JOIN v_sentiment_price_view p
        ON s.ticker = p.ticker AND s.event_date = p.transcript_date
    WHERE s.sentiment_polarity IS NOT NULL
)
SELECT
    CASE
        WHEN sentiment_polarity > 0.2 THEN 'POSITIVE'
        WHEN sentiment_polarity < -0.2 THEN 'NEGATIVE'
        ELSE 'NEUTRAL'
    END AS sentiment_bucket,
    COUNT(*) AS transcript_count,
    ROUND(AVG(next_1d_return), 6) AS avg_1d_return,
    ROUND(AVG(next_5d_return), 6) AS avg_5d_return,
    ROUND(STDDEV(next_1d_return), 6) AS std_1d_return,
    ROUND(AVG(sentiment_subjectivity), 4) AS avg_subjectivity
FROM sentiment_returns
GROUP BY sentiment_bucket;
```

**Columns:** `sentiment_bucket, transcript_count, avg_1d_return, avg_5d_return, std_1d_return, avg_subjectivity`

### 4.6 New View: v_ar1_time_series

**Purpose:** AR(1) autoregressive model estimated via OLS over rolling 60-day windows: `r_t = α + β · r_{t-1} + ε`. Beta (β) measures mean-reversion vs momentum; R² measures predictive power.

**SQL:**
```sql
CREATE OR REPLACE VIEW v_ar1_time_series AS
WITH return_series AS (
    SELECT
        ticker, date, close,
        (close - LAG(close, 1) OVER (PARTITION BY ticker ORDER BY date))
            / NULLIF(LAG(close, 1) OVER (PARTITION BY ticker ORDER BY date), 0) AS daily_return
    FROM silver_price
),
ar_input AS (
    SELECT
        ticker, date, close, daily_return,
        LAG(daily_return, 1) OVER (PARTITION BY ticker ORDER BY date) AS lag_return
    FROM return_series
),
ar_coeffs AS (
    SELECT
        ticker, date, close, daily_return, lag_return,
        REGR_SLOPE(daily_return, lag_return) AS beta_ar1,
        REGR_INTERCEPT(daily_return, lag_return) AS alpha_ar1,
        REGR_R2(daily_return, lag_return) AS r_squared_ar1,
        REGR_COUNT(daily_return, lag_return) AS n_obs
    FROM ar_input
    WHERE lag_return IS NOT NULL
)
SELECT
    ticker, date, close, daily_return,
    ROUND(alpha_ar1, 8) AS alpha_ar1,
    ROUND(beta_ar1, 8) AS beta_ar1,
    ROUND(r_squared_ar1, 6) AS r_squared_ar1,
    n_obs
FROM ar_coeffs
WHERE beta_ar1 IS NOT NULL AND n_obs >= 20
ORDER BY ticker, date;
```

> **Note on window syntax:** This version uses REGR_* as standard ordered-set aggregates without a custom `ROWS BETWEEN` frame, relying on DuckDB's default framing behavior. If the beta is unstable at the tail of the series, wrap in a subquery with `WHERE rn >= 20` using `ROW_NUMBER()` to require a minimum observation window.

**Columns:** `ticker, date, close, daily_return, alpha_ar1, beta_ar1, r_squared_ar1, n_obs`

**Interpretation for presentation:**
- `β ≈ 1`: Random walk (unit root); past returns do NOT predict future (weak-form market efficiency)
- `β ≈ 0`: No linear autocorrelation (white noise / uncorrelated returns)
- `|β| < 1`: Covariance-stationary AR(1); deviations decay exponentially over time
- `|β| > 1`: Explosive (non-stationary); not expected in daily equity returns
- `R²`: Fraction of variance explained by past return; meaningful only alongside N and β
- **Practical note:** In equity returns, β is typically close to 0 (daily returns are nearly unpredictable)
